# 网文AI工厂 — 项目说明

本文档说明项目背景、所采用的 CrewAI 框架、六 Agent 设计以及模型分配方案与原因。

---

## 一、项目背景

### 1.1 目标与定位

**网文AI工厂** 是一个面向 **中文网文创作** 的 AI 辅助写作工具。目标是通过多角色、流水线式的 AI 协作，在用户提供大纲的前提下，自动完成单章或整本书的正文生成，并兼顾：

- **结构清晰**：大纲拆解、伏笔与爽点规划；
- **人设一致**：人物卡与世界观维护，减少后期“崩人设”；
- **网文感**：节奏、爽点设计、语言风格贴近网文阅读习惯；
- **连贯与质量**：情节审查与最终润色。

### 1.2 技术选型

- **前端与交互**：Streamlit，支持多项目管理、大纲编辑、生成控制、导出与运行日志查看。
- **多 Agent 编排**：CrewAI，用于定义角色（Agent）、任务（Task）、执行流程（Crew）与记忆（Memory）。
- **大模型接入**：通过 OpenAI 兼容接口同时使用 **DeepSeek**、**通义千问（千问）**、**Kimi** 三家 API，按角色分工选用不同模型，兼顾能力与成本。

### 1.3 使用流程概览

1. 用户创建项目并编辑/粘贴大纲；
2. 选择「生成完整本书」或「续写后续章节」及章节数；
3. 系统会先根据大纲自动提炼一份《剧情圣经》（全局设定资产），后续按章依次执行 6 个 Agent 的流水线（见下文），并利用 CrewAI 记忆保持跨章上下文；
4. 生成结果保存为 Markdown，支持导出 TXT、Word、EPUB 等格式。

## 二、CrewAI 框架介绍与简单原理

### 2.1 什么是 CrewAI

CrewAI 是一个用于构建 **多智能体（Multi-Agent）** 工作流的开源框架。其核心思想是：把复杂任务拆成多个步骤，由多个具备不同「角色」与「目标」的 Agent 按顺序或按规则协作完成，每个 Agent 背后可以挂载不同的大模型（LLM）。

- 官网与文档：[https://www.crewai.com/](https://www.crewai.com/)
- 本项目依赖：`crewai>=0.75.0`

### 2.2 核心概念

| 概念 | 含义 |
|------|------|
| **Agent（智能体）** | 具有固定角色（role）、目标（goal）与背景故事（backstory）的“虚拟成员”，每次执行任务时由指定的 LLM 驱动。 |
| **Task（任务）** | 交给某个 Agent 的一项具体工作，包含任务描述（description）与期望输出（expected_output）。 |
| **Crew（班组）** | 由多个 Agent 和多个 Task 组成的整体；负责按某种流程（如顺序）执行任务并把输出在任务间传递。 |
| **Process（流程）** | 定义任务执行顺序。本项目使用 **sequential（顺序）**：Task1 → Task2 → … → Task6，前一个的输出作为后一个的上下文。 |
| **Memory（记忆）** | Crew 级别的跨任务、跨轮次记忆。可将关键信息（如人物、伏笔）向量化存储，在后续任务中按需召回，供 Agent 参考。 |

### 2.3 简单执行原理

1. **初始化**：根据当前章节号与用户大纲，创建 6 个 Agent 和 6 个 Task，并指定每个 Task 由哪个 Agent 执行。
2. **顺序执行**：Crew 按 Process.sequential 依次执行 Task1～Task6；每个 Task 的输入中会包含用户大纲、上一章摘要（若有）以及前面 Task 的输出（由框架在上下文中传递）。
3. **记忆**：若开启 `memory=True`，CrewAI 会在任务执行过程中将重要信息写入记忆（需配合 embedder 做向量检索）；在后续章节或后续任务中，可自动召回相关记忆，注入到 Agent 的上下文中。
4. **输出**：最后一个 Task（终极编辑润色师）的输出即为该章的最终正文，由项目保存为 `第N章.md`。

因此，**CrewAI 在本项目中 = 多角色流水线编排器 + 可选记忆层**，具体“写什么、怎么写”由各 Agent 所连接的 LLM 决定。

## 三、六 Agent 设计与任务流水线

### 3.1 流水线总览

每一章的生成都经过以下 5 步，顺序固定：

```
用户大纲 + 上一章摘要
        ↓
┌─────────────────────────────────────────────────────────────┐
│ 1. 大纲动态优化师   → 本章详细提纲、爽点位置、伏笔安排          │
│ 2. 人物与世界观守护者 → 人物卡、世界观设定（创建或更新）        │
│ 3. 爽点强化设计师   → 3 套爽点方案                           │
│ 4. 章节主写手      → 3500～5500 字正文                       │
│ 5. 终极审校专家     → 审查报告 + 润色后的最终章节（合并步骤）    │
└─────────────────────────────────────────────────────────────┘
        ↓
    第 N 章.md（并参与记忆与下一章的上文摘要）
```

### 3.2 各 Agent 角色与任务说明

| 顺序 | Agent 名称 | 角色定位 | 目标（goal） | 本步任务与产出 |
|------|------------|----------|-------------|----------------|
| 1 | **大纲动态优化师** | 资深网文编辑，擅长把创意扩展成长篇大纲 | 把用户大纲细化成每章详细提纲、爽点清单、伏笔表 | 根据用户大纲 + 上一章摘要，产出本章详细提纲，包含章节目标、主要情节、爽点位置、伏笔安排。 |
| 2 | **人物与世界观守护者** | 严谨的世界观与人物设定维护者 | 维护完整人物卡、世界观一致性，防止后期崩人设 | 基于上一步提纲，创建或更新人物卡与世界观设定文档。 |
| 3 | **爽点强化设计师** | 网文爽点设计专家 | 为每一章提供 3 套极简爽点方案 | 产出三个简短的爽点/看点列表。 |
| 4 | **章节主写手** | 高产网文作家，文笔流畅、代入感强 | 根据最新大纲 + 人物卡写出 3500～5500 字正文 | 综合前面所有准备，撰写本章完整正文。 |
| 5 | **终极审校专家** | 资深主编，兼具毒辣眼光与顶级文笔 | 一步完成情节逻辑审查与文字润色，输出最终定稿 | 快速检查伏笔/逻辑/人设（有硬伤直接改），并进行最终润色，产出最终章节定稿。 |

上一章内容会以「上一章内容摘要」（约 1000 字）的形式注入到各 Task 的描述中，用于保持章与章之间的连贯性；同时 CrewAI 记忆会存储人物、伏笔等信息，在后续章节中按需召回。

另外，本项目会在项目级别维护一份 **《剧情圣经》**（由系统从大纲提炼），其中包含世界观规则、人物卡、主线推进、伏笔表等“全局稳定信息”。生成每章时优先引用《剧情圣经》而非重复传入完整大纲，从而降低 token 成本并避免 embedding 查询超限。

## 四、模型分配方案与三种模式

本项目支持三种模型链路模式，以满足不同场景下的创作需求。您可以在系统设置中一键切换。

### 4.1 三种模式下的模型分配表

| 角色 (Agent) | **Speed (极速模式)** | **Balanced (平衡模式)** | **Quality (最高质量模式)** |
| :--- | :--- | :--- | :--- |
| **1. 大纲动态优化师** | DeepSeek-V3 (deepseek-chat) | DeepSeek-V3 (deepseek-chat) | **DeepSeek-R1** (deepseek-reasoner) |
| **2. 人物与世界观守护者** | Qwen-Plus | Qwen-Plus | **Kimi K2.5** |
| **3. 爽点强化设计师** | Qwen-Plus | Qwen-Plus | **Qwen-Max** (qwen-max) |
| **4. 章节主写手** | Qwen-Plus | Qwen-Plus (或 Kimi) | **Qwen-Max** (或 Kimi) |
| **5. 终极审校专家** | Qwen-Plus | Kimi K2.5 | **Qwen-Max** |

**模式说明：**
*   **Speed (极速模式)**：全链路优先使用响应最快、成本最低的模型（DeepSeek-V3 + Qwen-Plus），适合快速出稿、验证大纲或日更压力大的场景。
*   **Balanced (平衡模式)**：在核心写作环节引入 Kimi 进行润色，平衡速度与文笔细腻度。
*   **Quality (最高质量模式)**：全链路启用 2026 年旗舰模型（DeepSeek-R1 推理 + Kimi K2.5 长记忆 + Qwen-Max 顶级文笔），不计成本追求逻辑严密性与文学质感。

### 4.2 使用的模型与 API

本项目通过 **OpenAI 兼容接口** 使用三家国内大模型，并在代码中明确到具体模型名：

| 厂商 | API 模型名 | 对应产品/版本 | 上下文长度 |
|------|------------|----------------|------------|
| **DeepSeek** | `deepseek-chat` / `deepseek-reasoner` | V3 / R1 (推理版) | 128K |
| **通义千问** | `qwen-plus` / `qwen-max` | Plus / Max (旗舰版) | 128K / 32K |
| **Kimi** | `kimi-k2.5` | Kimi K2.5 | 256K |

- **DeepSeek**：`https://api.deepseek.com/v1`
- **通义千问**：`https://dashscope.aliyuncs.com/compatible-mode/v1`
- **Kimi**：`https://api.moonshot.cn/v1`

### 4.3 为什么这样分配（Quality 模式为例）

- **DeepSeek-R1 负责「大纲」**：利用 R1 满血版的强大推理能力，确保长篇剧情的因果链严密，逻辑零崩盘，适合做复杂的伏笔设计与结构规划。
- **Kimi K2.5 负责「设定守护」**：利用 Kimi 的百万字级超长上下文与记忆能力，精准锁定人设与世界观，杜绝后期“吃设定”。
- **Qwen-Max 负责「主写 + 审校」**：Qwen-Max 的中文网文语感、爽点节奏把控与金句爆发力目前最强，适合产出高质量正文并进行最终打磨。

### 4.4 记忆功能与 Embedding

- CrewAI 的 **记忆（Memory）** 用于在任务与章节之间保留人物、伏笔等关键信息；记忆的存储与检索依赖 **向量化（Embedding）**。
- 本项目将 **记忆的 Embedding** 单独配置为 **通义千问** 的文本向量接口（DashScope 的 `text-embedding-v3`，OpenAI 兼容），与上述三个 LLM 并存，无需额外 Key。
- 这样既保证记忆功能可用，又保持「中文网文」场景下检索语义一致；写书仍由 DeepSeek / 千问 / Kimi 三个 LLM 按上表分工完成。

### 4.5 为何「第一章」就会报 Embedding 查询超限？

- CrewAI 在做记忆检索时，会把 **当前任务的 `description` 与 `context` 拼成一段字符串**，作为「查询」发给 Embedding API，用于在向量库里做语义搜索。
- 我们的第一个任务（大纲动态优化师）的 `description` 里包含 **完整用户大纲**。只要大纲较长（例如超过约 4000 字），这段字符串在千问侧会超过 **text-embedding-v3 的单次输入上限 8192 token**，于是返回 `Range of input length should be [1, 8192]`。
- 因此 **不是「写了很多章才超」**，而是 **每次执行任务前做记忆召回时，用来检索的那段「任务描述 + 上下文」本身就超长了**，第一章第一个任务就会触发。
- **处理方式**：在 `src/tasks.py` 里对传入任务描述的大纲做安全截断（约 3800 字），使「任务描述 + 上文摘要」总长落在 Embedding 限制内，记忆检索即可正常进行。若大纲超过该长度，首步任务会看到截断后的大纲；建议单次大纲控制在约 4000 字以内，或接受首步使用截断版本。

## 五、项目结构速览

```
AI_novel/
├── app.py                 # Streamlit 入口，多项目、大纲、生成、导出、日志
├── 项目说明.md             # 本文档
├── src/
│   ├── agents.py          # 6 个 Agent 定义及 DeepSeek/千问/Kimi 的 LLM 配置
│   ├── tasks.py           # 6 个 Task 定义及与 Agent 的绑定
│   ├── generator.py       # Crew 组装、记忆与 embedder 配置、按章生成逻辑
│   ├── project.py         # 项目/章节/大纲的存储与加载
│   ├── api.py             # API Key 加密存储与多模型测试
│   ├── export.py          # 导出 TXT / Word / EPUB
│   └── logger.py          # 运行日志
├── projects/              # 各小说项目目录（大纲、章节、日志等）
└── .crewai/               # CrewAI 记忆与数据目录（若启用 memory）
```
