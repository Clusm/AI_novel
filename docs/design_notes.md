
# 项目思考与复盘：从卡顿到极速生成的进化之路

本文档记录了《AI_novel》项目在开发过程中遇到的核心问题、解决方案以及架构取舍，旨在为后续优化和类似项目提供经验参考。

## 一、 核心复盘：那些反复折磨我们的“卡顿”与“不保存”

在开发过程中，我们遇到最棘手的问题主要集中在 **“生成过程卡死”** 和 **“终审完成后不保存”** 这两个现象上。

### 1. 生成过程长时间无响应（假死）
**现象**：点击生成后，界面长时间无反应，或者进度条不动，终端日志停滞。
**原因分析**：
- **DeepSeek-R1 (Reasoner) 思考过长**：推理模型在规划大纲时容易陷入无限思维链，且 Streamlit 是同步执行，主线程被阻塞，导致前端看起来像死机。
- **Agent 缺乏超时限制**：早期的 Agent 配置中漏掉了 `max_execution_time`，一旦模型响应慢或网络波动，Agent 会一直死等。
- **Windows 终端挂起**：在 Windows CMD/PowerShell 中，鼠标点击窗口可能触发“快速编辑模式”，导致整个进程被系统挂起。

**解决方案（多重保活）**：
- **强制超时**：给所有 Agent（特别是大纲和爽点）增加 `max_execution_time`（如 600s）和 `max_iter`（如 3次）限制，防止无限等待。
- **心跳日志**：在 `step_callback` 中增加实时日志输出，让用户明确感知后台仍在运行。
- **非阻塞轮询**：修改执行逻辑，不再死等 `crew.kickoff()` 返回，而是轮询 Task 输出。

### 2. 终极审校后内容未保存（丢稿）
**现象**：终极审校 Agent 跑完了，控制台也显示完成了，但文件夹里没有生成 md 文件。终端常伴随 `Tracing is disabled` 提示。
**原因分析**：
- **CrewAI 生命周期阻塞**：`crew.kickoff()` 在任务结束后会执行 Memory/Knowledge 保存和 Telemetry 上报。如果网络不通或文件被锁，主线程会卡在这里，导致后面的 `save_chapter` 代码无法执行。
- **返回值解析错误**：`kickoff` 返回的 `result` 对象在某些情况下不是纯文本，直接保存可能为空。
- **Tracing 误导**：`Tracing is disabled` 是 CrewAI 的默认面板输出，虽非报错，但它出现在卡顿点，掩盖了真正的阻塞原因（Telemetry/Memory）。

**解决方案（防御性编程）**：
- **暴力禁用 Tracing/Telemetry**：在代码入口处通过 `os.environ` 强制禁用所有 CrewAI 和 OpenTelemetry 的联网功能。
- **实时兜底保存**：在 `step_callback` 回调中增加“实时保存”机制，一旦检测到“终极审校专家”输出有效文本（>1000字），立即写入临时文件。
- **绕过 Kickoff**：主流程直接从 `tasks[-1].output.raw` 获取结果，不再依赖 `kickoff` 的完整生命周期结束。

## 二、 架构取舍：为了稳定与速度，我们放弃了什么

为了解决上述问题，我们对系统架构进行了多次“做减法”的取舍。

### 1. Agent 职能合并（做减法）
- **问题**：原流程中“爽点强化设计师”单独跑一遍，浪费时间且容易与大纲逻辑冲突。
- **取舍**：**砍掉“爽点强化设计师” Agent**。
- **替代方案**：在“大纲优化师”的 Prompt 中强制要求：生成细纲时，**直接在每个情节段落后标注【爽点/期待感】**。
- **收益**：减少一次 LLM 调用，每章生成速度提升 30-60 秒，且逻辑更自洽。

### 2. 记忆功能降级（保稳定）
- **问题**：开启 CrewAI 内置 Memory 会导致生成结束后长时间阻塞（向量化/保存），且容易引发“不保存”的 Bug。
- **取舍**：**默认关闭 CrewAI Memory**。
- **替代方案**：使用 **“剧情圣经” + “章节摘要链”** 的显式上下文传递。
  - 每章生成后自动生成 `第N章摘要.md`（强制限制 900 字）。
  - 下一章生成时，显式读取上一章摘要和圣经作为 Context。
- **收益**：彻底解决了收尾阶段的卡顿问题，且上下文控制更透明、可审查。

### 3. 模型分层策略（保质量与速度）
- **取舍**：
  - **大纲/爽点/人物**：使用 Qwen/DeepSeek V3（速度快，逻辑够用）。
  - **正文写作**：使用 Qwen-Max（长文生成能力强）。
  - **终极审校**：**强制使用 Kimi K2.5**（如果 Key 存在），无论前端选什么模式，最后一道关卡必须用最好的模型来润色，保证文字质感。

## 三、 协作建议：如何更高效地与 AI 结对编程

作为您的 AI 结对程序员，我非常享受与您共同解决问题的过程。为了在未来更高效地解决问题，我有以下几点建议：

### 1. 提供具体的报错“上下文”
- **好的反馈**：“生成到第二章终审后卡住，终端最后一行显示 `Tracing is disabled`，文件夹里只有临时文件。”
- **解释**：这能让我迅速定位是“流程阻塞”还是“逻辑错误”。

### 2. 明确“预期”与“现状”的差异
- **好的反馈**：“我关闭了记忆功能，担心文本质量下降。有没有不开启 Memory 但能保持连贯性的方案？”
- **解释**：这能让我跳出单纯的“修复 Bug”模式，为您提供架构级的替代方案（如章节摘要链）。

### 3. 反馈修复效果
- **好的反馈**：“刚才的修改有效，第二章保存了。但是现在第三章大纲生成又变慢了。”
- **解释**：及时的正向/负向反馈能让我调整优化策略，避免在错误的路径上越走越远。

### 4. 敢于质疑与提出“笨办法”
- **例子**：“不要写代码，请全面思考整个流程...有无别的优化方案？”
- **解释**：有时候代码修补只是掩盖问题，架构重构（如合并 Agent）才是治本之策。您的大局观提问非常关键。

## 四、 近期技术攻坚：多线程死锁与打包发布

在项目从脚本迈向桌面应用（EXE）的过程中，我们经历了多次“试错-定位-修正”的循环，以下是两个最具代表性的技术攻坚案例。

### 1. 多线程死锁导致的“终极卡死”
**现象**：生成到了最后一步（终极审校），控制台有输出，但界面进度条不动，文件未保存，程序彻底假死。
**反复排查过程**：
1.  **初判**：以为是 CrewAI 的 `kickoff` 没有返回结果，尝试增加超时和强制结束 -> **无效**。
2.  **再判**：以为是网络问题导致 API 挂起，增加重试机制 -> **无效**。
3.  **终判**：发现 CrewAI 的 Agent 运行在后台线程，而 Streamlit 的界面更新（`st.write`, `st.empty`）必须在主线程执行。后台线程直接调用 UI 更新函数引发了 **线程死锁（Deadlock）**。

**最终方案（生产者-消费者模式）**：
- **引入队列（Queue）**：在 `generator.py` 中创建一个线程安全的 `queue.Queue`。
- **解耦**：Agent（生产者）只把日志 `put` 进队列；Streamlit 主循环（消费者）从队列 `get` 消息并更新 UI。
- **结果**：彻底解决了“卡在99%”的问题，界面流畅且不再锁死。

### 2. PyInstaller 打包的“依赖地狱”与“白屏”
**现象**：打包出的 EXE 要么运行报错，要么双击后只有一个白屏，什么都不显示。
**反复迭代过程**：
1.  **编码错误**：Batch 脚本输出中文导致乱码报错 -> **改为全英文输出**。
2.  **缺失依赖**：报错 `ValueError: Unknown encoding cl100k_base` -> **PyInstaller 漏掉了 `tiktoken` 的数据文件**。
    - **修复**：在打包命令中增加 `--collect-all tiktoken` 和 `--collect-all litellm`。
3.  **白屏无响应**：程序启动了 Streamlit，但没有浏览器弹窗，且默认端口 8501 被占用时直接静默失败。
    - **修复**：编写 `run_app.py` 启动器，实现 **“动态查找可用端口”** + **“后台线程自动唤起浏览器”**。

**协作启示**：
- **环境隔离**：打包问题往往与本地环境有关，使用 `.venv` 并在脚本中显式激活是关键。
- **全量日志**：打包失败时，完整的 traceback（堆栈跟踪）比“报错了”更有价值，比如 `cl100k_base` 这个关键词直接指向了 `tiktoken` 库。

## 五、 近期GUI重构与工程化复盘 (2025-03-11)

### 1. 核心问题与反复迭代

#### (1) GUI 视觉的“现代感”定义偏差
*   **问题**：用户多次提出“界面不够美观”、“需要现代产品级效果”，但初期交付仅停留在“工整”层面。
*   **反复过程**：
    1.  **第一版**：使用原生 Qt 控件，仅做简单布局 -> 用户反馈“不够自然”。
    2.  **第二版**：增加圆角和边框 -> 用户反馈“左上角文字多余”、“颜色不统一”。
    3.  **第三版（最终版）**：彻底去除了侧边栏边框，统一了背景色（#f8fafc），引入了阴影悬浮卡片设计，使用了更细腻的字体栈（Segoe UI/Microsoft YaHei UI），并增加了语义化的颜色条。
*   **原因**：开发者视角（功能优先）与产品视角（体验优先）的错位。“现代感”在技术实现上对应的是：**无边框设计（Borderless）、微交互（Hover effects）、层级感（Shadows）和留白（Spacing）**。
*   **解决方案**：
    *   **重写 QSS**：放弃原生边框，改用 `QGraphicsDropShadowEffect` 营造层级。
    *   **布局重构**：移除 `QTabWidget` 的原生边框，隐藏 `QSplitter` 把手，使界面浑然一体。
    *   **语义化增强**：日志流从纯文本改为 HTML 渲染，增加颜色和图标，模拟“黑客终端”体验。

#### (2) 依赖环境的“隐形墙”
*   **问题**：点击生成直接报错 `ModuleNotFoundError: No module named 'crewai'`。
*   **原因**：
    *   项目依赖未完全安装或环境切换导致。
    *   国内网络环境下，`pip install crewai` 因 SSL/连接超时反复失败。
*   **解决方案**：
    *   明确使用清华镜像源：`-i https://pypi.tuna.tsinghua.edu.cn/simple`。
    *   在代码中增加防御性检查（`try-import`），虽然本次是直接修复环境，但未来应在启动时自检。

#### (3) 功能与交互的取舍
*   **问题**：大纲编辑与预览分离，操作繁琐。
*   **取舍**：放弃了“左侧编辑、右侧预览”的标准 Markdown 编辑器模式。
*   **决策**：合并为一个“所见即所得”风格的编辑器。虽 `QTextEdit` 对 Markdown 支持有限，但通过 `setMarkdown` 和自定义样式，满足了“输入即渲染”的核心诉求，减少了用户的认知负担。

### 2. 协作建议：如何更高效地探讨解决问题

为了减少“猜谜式”迭代，建议在未来的协作中：

1.  **提供视觉参照（Visual Reference）**：
    *   **建议**：当提到“现代感”或“好看”时，如果能给出一个类似的软件名称（如 Notion、Obsidian、VS Code）或一张截图，我们能一步到位。
2.  **明确“不要什么”（Negative Constraints）**：
    *   **建议**：像这次您提到的“不要左上角的文字”、“不要原本的边框颜色”，这种**具体的否定指令**非常高效，能帮助我快速定位并剔除视觉噪音。
3.  **场景化描述需求**：
    *   **建议**：描述功能时带上场景。例如：“我需要在运行生成时，像看黑客帝国屏幕一样看到后台在做什么”，这比“优化日志显示”更能激发我实现“HTML彩色日志流”的灵感。
