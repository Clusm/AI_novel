# 一致性与可控（CONSISTENCY）

本项目的一致性目标是：跨章节不“打架”，让人设、设定、伏笔、节奏在长期连载中保持稳定。

## 为什么会跑偏

- 单章写作信息量大，模型容易把“本章爽点”优先级抬高，忽略跨章约束
- 长大纲无法完整注入每章上下文，导致后续章“看不到细纲”
- 仅靠“上一章末几句”很难稳定承接状态与伏笔

## 本项目的强一致性资产（必须理解）

- 剧情圣经：从大纲提炼的全局稳定信息（世界观铁律/人物卡/伏笔表/禁忌）
- 章节摘要：每章保存为 `summaries/第N章摘要.md`，作为下一章强输入
- 事实台账：每章保存为 `canon/第N章台账.md`，沉淀不可逆事实与锚点

这三类资产共同解决“长篇记忆”与“跨章承接”的问题。

## 关键机制与默认策略

### 1）按章节号抽取大纲片段

系统会优先在 `大纲.md` 中定位 `第N章` 对应段落作为当前章参考，而不是只读取大纲开头，从而让“写长大纲不浪费”。

### 2）终审一致性 Gate（先验收再润色）

终审必须先检查并修正硬伤，再润色输出定稿，重点关注：

- 承接上一章末状态（并在本章前 20% 篇幅落地）
- 不新增未铺垫核心设定
- 不出现角色状态回滚/瞬移/突然知道未出现信息

### 3）章节内情节标题去除

成品正文会自动去掉章节内 `##/###` 分段标题及“一、二、三”式小标题，只保留：

```text
# 第N章 标题
正文...
```

### 4）正文长度下限保障

默认要求正文（不含摘要块）≥3500 字，不足会在保存前触发自动扩写补足（不改剧情顺序与结果）。

### 5）写手摘要块（被剥离保存）

写手会在正文末尾输出摘要块：

```text
[SUMMARY_BEGIN]
...
[SUMMARY_END]
```

终审与保存阶段会剥离该区块并单独保存到 `summaries/`，避免污染正文，同时提高摘要质量。

## 大纲写法建议（最关键的“可控输入”）

- 开头 800–1200 字：集中写硬约束（系统铁律/人设底线/节奏铁律/禁忌红线）
- 章节细纲尽量使用 `第N章` 标题行
- 伏笔尽量用“伏笔名 + 回收条件/回收章位”的形式落表
