# 技术工作原理（多 Agent 协同写作）

本文档从「数据流」「任务编排」「记忆系统」「大纲利用」「关键技术栈」五个角度，解释网文AI工厂在代码层面是如何把用户大纲变成可连贯扩展的章节正文的。

---

## 1. 系统由哪些模块组成

项目核心可以理解为三层：

1) **交互层（UI）**：`app.py`（Streamlit）  
2) **编排层（多 Agent 工作流）**：`src/generator.py` + `src/agents.py` + `src/tasks.py`（CrewAI）  
3) **资产层（项目文件与记忆存储）**：`src/project.py` + `.crewai/` + `projects/`（本地文件 + CrewAI 记忆数据库）

它们之间的主数据流是：

```text
用户在 UI 中输入/粘贴大纲
  ↓（保存到 projects/{项目名}/大纲.md）
UI 点击“一键生成”
  ↓（把大纲文本作为参数传入）
src/generator.py 组装 6 个 Agent + 6 个 Task
  ↓（顺序执行 Process.sequential）
CrewAI 依次驱动 LLM 完成：提纲→设定→爽点→正文→审查→润色
  ↓（保存到 projects/{项目名}/chapters/第N章.md）
并将关键内容写入记忆（.crewai/data/）
```

## 2. 多个 Agent 是如何协同工作的

### 2.1 协同机制 = 顺序流水线 + 上下文传递 + 可检索记忆

本项目使用 CrewAI 的顺序流程：

- 代码入口：`src/generator.py` 中 `Crew(process=Process.sequential, ...)`
- 任务定义：`src/tasks.py` 中 `create_tasks(...)` 生成 6 个 Task
- 角色定义：`src/agents.py` 中 `create_agents()` 生成 6 个 Agent（绑定不同模型）

关键点在于：

1) **同一章内的协同**  
   `Task1 → Task2 → ... → Task6` 按顺序执行；前面任务的输出会被 CrewAI 作为上下文传递给后续任务。  
   这使得后续 Agent 不需要“重新理解一遍世界”，只要在前序输出的基础上继续加工即可。

2) **跨章节的协同（长期一致性）**  
   `memory=True` 时，CrewAI 会把执行过程中出现的关键信息向量化写入记忆库；下一次任务执行前，会把「任务 description + context」拼成检索 query，召回相关记忆并注入上下文，让后续章节能复用人物设定、伏笔等信息。

### 2.2 六个 Agent 的分工与串联

每一章固定走 6 个角色（对应 `src/agents.py`）：

1) 大纲动态优化师（DeepSeek）：把“用户大纲 + 全局设定”落到本章的详细提纲  
2) 人物与世界观守护者（千问）：基于提纲更新人物卡/世界观一致性  
3) 爽点强化设计师（Kimi）：为本章设计多套爽点并选最优  
4) 章节主写手（千问）：按提纲与爽点写出正文  
5) 情节连贯审查员（千问）：从逻辑/伏笔/OOC 角度给出审查  
6) 终极编辑润色师（Kimi）：最终语言与节奏打磨，输出成稿

简化成“协作链”就是：

```text
骨架（提纲）→ 设定护栏（人物/世界观）→ 强刺激（爽点）→ 产出正文 → 质检（审查）→ 成稿（润色）
```

这种设计的工程意义是：

- 把“不可控的长文本生成”拆成多个可控步骤，降低单次失败成本
- 把一致性问题（人物/世界观）前置，减少后期返工
- 把质量保障（审查/润色）放到写完之后，形成闭环

## 3. 用户输入的大纲是如何被运用的

大纲在项目中不是“只喂给一次模型”，而是同时承担两种职责：

1) **全局设定来源（提炼成剧情圣经）**：用于跨章节一致性  
2) **章节规划依据（大纲摘录注入任务）**：用于当前章的局部决策

### 3.1 大纲的落盘与读取

用户在 UI 中编辑/粘贴的大纲会保存到：

- `projects/{项目名}/大纲.md`（见 `src/project.py` 的 `save_outline` / `load_outline`）

生成时 UI 把大纲全文字符串直接传入生成器：

- `app.py` 调用 `generate_single_chapter(selected_project, outline_content, chapter_num, ...)`

### 3.2 为什么要先生成《剧情圣经》

在 `src/generator.py` 中，`ensure_story_bible(project_name, outline)` 会在以下情况下触发“从大纲提炼剧情圣经”：

- 项目没有剧情圣经；或
- 大纲发生变化（通过 SHA256 哈希检测）

剧情圣经保存到：

- `projects/{项目名}/剧情圣经.md`（见 `src/project.py` 的 `save_story_bible` / `load_story_bible`）

这样做的目的主要有两点：

1) **减少每章重复传入完整大纲的 token 成本**  
   大纲通常很长，但其中只有一部分是“长期稳定设定”（世界观规则、人设、主线、伏笔表等）。把这部分提炼成剧情圣经，后续每章优先引用它，能显著降低上下文体积。

2) **配合记忆检索的输入长度限制**  
   CrewAI 记忆检索会把「任务 description + context」作为 query 发给 Embedding 接口；如果 description 里塞入完整大纲，容易超出 embedding 单次输入上限。  
   因此项目在 `src/tasks.py` 中对“剧情圣经摘要”和“大纲摘录”做了长度限制（`STORY_BIBLE_MAX_CHARS`、`OUTLINE_EXCERPT_MAX_CHARS`）。

### 3.3 大纲如何进入每个 Agent 的任务

每章创建任务时（`src/tasks.py:create_tasks`）会做两件事：

1) 把剧情圣经（摘要）拼入 `base_context`  
2) 把用户大纲（摘录）拼入 `base_context`

然后：

- Task1（大纲动态优化师）在 description 中明确要求“为第 N 章生成详细提纲”，并附带上述上下文  
- 后续 Task 的 description 更短（例如“基于前面的提纲更新人物卡”），但会通过 CrewAI 的上下文传递拿到前序输出，同时也能通过记忆召回拿到历史设定

换句话说：  
大纲既会“直接影响本章提纲”，也会“间接影响后续写作与审查”，因为它被压缩成剧情圣经与记忆资产，持续参与后续章节。

## 4. 记忆系统是怎么工作的（为什么能跨章保持一致）

### 4.1 记忆启用点与向量化配置

章节生成时启用记忆：

- `src/generator.py` 中创建 Crew：`memory=True`

同时配置 embedder（用于向量化检索）：

- `src/generator.py:get_embedder_config()` 使用 DashScope 的 OpenAI 兼容 embedding：`text-embedding-v3`

### 4.2 记忆数据存储在哪里

`app.py` 在启动时把 CrewAI 存储目录固定到项目内：

- `CREWAI_STORAGE_DIR = <项目根目录>/.crewai/data`

因此向量库与运行输出会落在：

```text
.crewai/
  data/
    chroma.sqlite3
    long_term_memory_storage.db
    latest_kickoff_task_outputs.db
    ...
```

工程上这意味着：

- 记忆是“可持久化”的，不依赖 Streamlit 会话状态
- 同一个项目在多次运行之间仍可复用历史记忆

## 5. 主要技术与关键实现点

### 5.1 多 Agent 编排：CrewAI

- Agent：定义角色、目标、背景与所用 LLM（`src/agents.py`）
- Task：定义每步要做什么与期望输出（`src/tasks.py`）
- Crew：把 Agent 与 Task 组装成可执行流水线，并可开启记忆（`src/generator.py`）

### 5.2 多模型接入：OpenAI 兼容接口 + CrewAI LLM

本项目通过 OpenAI 兼容协议分别接入：

- DeepSeek：`https://api.deepseek.com/v1`（`openai/deepseek-chat`）
- 通义千问 DashScope：`https://dashscope.aliyuncs.com/compatible-mode/v1`（`openai/qwen-plus`）
- Kimi Moonshot：`https://api.moonshot.cn/v1`（`openai/kimi-k2.5`）

模型绑定发生在 `src/agents.py`，每个 Agent 固定使用自己的 LLM，从而让“结构规划 / 写作 / 润色”各自发挥长处。

### 5.3 API Key 安全存储：cryptography(Fernet)

`src/api.py` 将 3 个模型的 Key 加密保存到：

- `.api_keys.enc`（密文）
- `.encryption_key`（本地对称密钥）

Streamlit UI 通过 `src/api.py` 读写密钥，并用 `litellm` 做连接测试（避免把 Key 明文写入代码）。

### 5.4 导出：TXT / Word / EPUB

导出模块在 `src/export.py`，依赖（见 `pyproject.toml`）包括：

- `python-docx`：生成 Word
- `ebooklib`：生成 EPUB
- `pypandoc`：格式转换辅助（依赖系统 pandoc 时按环境而定）

---

## 6. 一句话总结

网文AI工厂的工作原理可以概括为：  
**把用户大纲提炼成可复用的“剧情圣经”，再用 CrewAI 把 6 个专职角色按顺序串成流水线，并用向量记忆让设定与伏笔跨章节持续可检索复用。**
