# 可控与一致性维护（原理与写法规范）

本文档解释本项目如何在“多模型 + 多 Agent + 长篇连载”场景下实现可控输出与跨章节一致性，并给出一套直接可用的写法规范，帮助你在不把大纲写得过长的前提下，把关键约束稳定传递给流水线。

---

## 1. 什么叫“可控”和“一致性”

### 1.1 可控（Control）

可控不是“模型不出错”，而是把输出收敛到你期望的范围内，主要包括：

- **剧情可控**：本章目标、冲突、反转、钩子能按计划出现
- **节奏可控**：爽点密度、信息揭示速度、战斗/日常比例稳定
- **风格可控**：叙述视角、系统提示格式、网感与尺度一致
- **成本可控**：上下文长度可预期，不因大纲过长导致检索或调用不稳定

### 1.2 一致性（Consistency）

一致性强调“跨章不打架”：

- **设定一致**：世界规则、境界体系、系统机制不自相矛盾
- **人设一致**：人物动机、底线、能力成长路径不跳变
- **伏笔一致**：埋点/回收能对上，关键道具/关系不丢失

---

## 2. 项目层面的可控与一致性机制

### 2.1 将长文本生成拆成 6 步：用流程控制质量

本项目不是让一个模型“一口气写完一章”，而是用顺序流水线把不确定性分摊到多个步骤：

1) 先产出本章提纲（骨架）  
2) 再做人物/世界观一致性约束（护栏）  
3) 再做爽点方案（刺激点可控）  
4) 再写正文（在前面约束下产出）  
5) 再审查（发现矛盾/漏洞）  
6) 最后润色（风格与节奏统一）

这属于“先规划、再生成、再质检”的工程化写作方式：把可控点前移，减少写完再返工。

### 2.2 用《剧情圣经》承载“全局稳定信息”

长篇最容易跑偏的不是单章细节，而是全局规则与人物关系。项目将这部分从大纲中提炼成《剧情圣经》，作为全书统一引用的资产：

- 世界观规则、系统铁律、境界体系
- 核心人物卡与关系网
- 爽点配方与节奏铁律
- 伏笔与回收表（以表格形式沉淀）
- 写作约束与禁忌红线

这样做的效果是：即便你不把所有内容塞进每章上下文，流水线仍可通过剧情圣经保持一致性。

### 2.3 用“长度控制”保证稳定传输与记忆检索可用

本项目每章创建任务时，会对“剧情圣经摘要”和“大纲摘录”做硬截断（在 `src/tasks.py` 中定义长度常量），避免两类问题：

- **调用成本不可控**：上下文暴涨导致每章成本飙升
- **记忆检索不稳定**：CrewAI 记忆检索会把 `description + context` 作为 embedding 查询输入，过长会触发输入长度限制

因此，大纲再长也不会导致“发不出去”，但会带来一个现实约束：每章直接注入的“大纲摘录”是有限长度的。

### 2.4 大纲与剧情圣经在工作流程中的实际应用方式

在工程实现上，大纲与剧情圣经分别承担“输入来源”和“稳定资产”的角色，并在章节生成流程里以不同方式被使用。

大纲与剧情圣经的来源与落盘：

- 用户在 UI 中编辑/粘贴的大纲保存到 `projects/{项目名}/大纲.md`（`src/project.py: save_outline/load_outline`）
- 系统会根据大纲生成/更新 `projects/{项目名}/剧情圣经.md`（`src/generator.py: ensure_story_bible`）

生成每一章时（从 `app.py` 触发到 `src/generator.py`）会发生三步关键注入：

1) 提炼或复用剧情圣经（全局一致性资产）  
   - 当大纲内容变化（哈希变化）或剧情圣经为空时，才会触发重新提炼；否则复用旧圣经。  
   - 这一步的目标是把“长大纲里真正稳定且跨章必须一致的信息”抽出来，变成可反复引用的设定底座。

2) 组装每章任务上下文（每章可见的输入窗口）  
   - 任务上下文会包含：剧情圣经摘要 + 用户大纲摘录 + 上一章摘要（若提供）。  
   - 其中剧情圣经与大纲都会被控制长度，避免上下文过长导致成本不可控或记忆检索输入超限。

3) 通过顺序流水线把输出逐步“收敛”  
   - Task1 先把“大纲/圣经/上一章摘要”落到本章提纲  
   - 后续任务复用前序输出，继续做一致性约束、爽点、正文、审查、润色  
   - 这相当于把“大纲”从一次性输入，变成了贯穿全章的控制信号

### 2.5 用记忆（memory=True）做“跨章召回”

记忆层的价值是把历史章节/任务中出现的关键点变成可检索资产：

- 人物新信息（新技能、新关系、新底线触发）
- 伏笔埋点（何时埋、怎么埋）
- 设定补丁（修正了什么规则、补充了什么限制）

后续章节在执行任务前，会先做语义检索召回相关记忆，把它注入上下文，从而降低“忘设定”的概率。

## 3. 为什么“只写 1200 字分卷核心剧情”会变得不够可控

如果把所有信息压缩到非常短的卷级摘要里，系统确实能稳定传输，但会出现典型退化：

- **章级决策素材不足**：提纲 Agent 很难稳定产出差异化章节结构，容易同质化或“水”
- **伏笔难落点**：没有明确“埋点章位/埋法”，后续回收难对齐
- **人物弧不连续**：只有卷级结论，缺少人物在关键节点的动机变化依据

因此更稳的策略是“两层写法”：用短摘要保证每章可见的硬约束，用长资料支撑剧情圣经与记忆资产沉淀。

---

## 4. 推荐的“两层大纲”写法（强可控、不过长）

### 4.1 第一层：前 800–1200 字“硬约束摘要”

目标：保证每章任务一定能读到的内容只放“必须每章都遵守”的东西。

建议字段（按优先级）：

- 系统铁律（结算时机、奖励/惩罚、不可绕过的规则）
- 世界观与境界体系一句话版本（避免每章口径变化）
- 主线阶段目标（每卷一句话 + 终局一句话）
- 人设底线（主角/核心女主/主要反派各 1–3 条）
- 写作约束（视角、人称、系统提示格式、禁忌红线）
- 节奏铁律（每几章一次小爽点/大冲突/格局变化）
- 伏笔总表摘要（只列“伏笔名 + 回收条件”）

### 4.2 第二层：后续“可沉淀资产区”

这些内容不要求每章都直接注入，但强烈建议写全，用来支撑剧情圣经与长期一致性：

- 人物卡完整版（动机/能力/雷区/事业线）
- 关系网表
- 名场面池（可复用的桥段与升级版）
- 伏笔与回收表（含埋点章节与状态）
- 分卷细化（关键事件列表、阶段钩子）
- 开局 10–20 章细纲（可控开局）

### 4.3 按章节号抽取“大纲片段”：写长也不浪费的关键机制

为了解决“每章只看到大纲开头的一小段，后面章节细纲喂不到”的问题，本项目在任务创建时改为按章节号抽取相关片段，再与开头硬约束拼接，最终仍控制在固定长度内。

抽取与拼接的思路（对应 `src/tasks.py` 内的 `_extract_outline_excerpt`）：

- 固定保留一段大纲开头作为“硬约束摘要”（用于保证每章都能读到系统铁律、人设底线、节奏与红线）
- 再从大纲中定位当前章节号对应的段落作为“章节相关片段”
- 两段拼接后做硬截断，保证总长度不超过 `OUTLINE_EXCERPT_MAX_CHARS`

定位规则（从稳定到兜底）：

1) 优先识别章节标题行：`第N章`  
   - 例如：`第1章：xxx`、`第 1 章：xxx`  
   - 命中后，抽取“本章标题到下一章标题之间”的整段内容

2) 如果大纲里没有规范的 `第N章` 标题，则使用宽松兜底匹配：  
   - `第N章` / `章节N` / `Chapter N`  
   - 命中后在附近截取一段窗口作为相关片段

3) 如果依然找不到，则退化为“只截断大纲开头”，保证系统仍可运行

对写法的直接建议：

- 把“必须每章都遵守的硬约束”集中写在大纲最前面 800–1200 字
- 章节细纲尽量用 `第N章` 作为标题行（这是抽取最稳定的锚点）
- 不想写全章细纲时，也至少写“每卷锚点章位”（第几章附近发生什么），让提纲 Agent 有落点

---

## 5. 章级可控的“最低成本锚点”（不写全章细纲也能稳）

如果你不想写每章细纲，可以只写“锚点章位”，让提纲 Agent 有明确落点：

- **每卷 5 个章位锚点**（用“第X章附近”表达即可）  
  1) 开局立规矩（系统/世界观/目标立住）  
  2) 第一波爽点爆发（第一次大奖励/打脸）  
  3) 卷中反转（规则被挑战/关系被改写）  
  4) 卷末高潮（势力格局变化/大敌登场）  
  5) 卷末钩子（下一卷核心矛盾抛出）

写法示例：

```text
第一卷锚点：
- 第1-3章：系统规则立住 + 主角底线立住
- 第8-10章：第一次大打脸 + 体系升级
- 第20章附近：关键女主入场并绑定长期矛盾
- 第45章附近：卷中反转（主角策略暴露/规则反噬）
- 第60章：卷末高潮 + 下一卷主敌钩子
```

这类锚点信息非常短，但对“剧情可控”提升显著。

## 6. 一致性维护的检查清单（写给审查员，也写给你）

每章生成后，至少用以下维度做一致性自检（你也可以把它作为审查员的硬指标）：

- **规则一致**：本章系统结算/境界描述是否与圣经一致
- **能力一致**：人物实力是否越级，是否缺失过渡
- **动机一致**：人物行为是否符合其动机与雷区
- **名词一致**：势力、地名、法宝、称谓是否统一写法
- **伏笔一致**：是否新增埋点，是否更新状态，是否破坏回收条件
- **节奏一致**：是否满足“每X章爽点/每Y章大冲突”的节奏铁律

---

## 7. 你可以怎么用这份文档

- 写大纲时：先把“硬约束摘要”写满，再补“资产区”
- 改大纲时：优先更新“硬约束摘要”和“伏笔表状态”
- 生成时跑偏：先检查“硬约束摘要是否缺失关键规则”，再看剧情圣经是否已更新
